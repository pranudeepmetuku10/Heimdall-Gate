Real-time data engineering reference

Heimdall-Gate

A streaming gateway that ingests public business-listing events, validates them at the edge, quarantines anything malformed, and serves trusted, analytics-ready Delta tables. Yelp Fusion → Kafka → Spark Structured Streaming → Delta Lake on Databricks.

Apache Kafka Spark Structured Streaming Delta Lake Databricks trigger(availableNow)
01

End-to-end flow

One responsibility per stage, an explicit contract at every boundary, and nothing reaches back upstream. The dashed red paths are the two quarantine sinks — no record is ever silently dropped.

02

The medallion, in detail

Three Delta layers, each idempotent. Bronze lands raw and never blocks on schema drift. Silver conforms and is correct under at-least-once delivery. Gold recomputes a bounded window so late data is reflected without rewriting history.

03

Technology at each layer

What runs where, and the one-line reason it was chosen over the obvious alternative.

Source

External system of record
Yelp Fusion API REST · Bearer

Why: a legally usable public feed of business metadata. Read-only; retains only trend-relevant fields per Yelp's ToU.

Ingestion

Transport & normalization
Python 3.11+ httpx tenacity pydantic v2

Why: pydantic gives a typed, validated envelope without a schema-registry service; tenacity isolates the retry/backoff policy from business logic.

Transport

Durable buffer & replay log
Apache Kafka confluent-kafka KRaft (local)

Why: librdkafka-backed client gives a true idempotent producer and real flush semantics. Kafka decouples the 5k/day API cap from downstream processing.

Processing

Validation & modeling
Spark Structured Streaming availableNow Databricks Serverless

Why: the same streaming engine and checkpoint protocol as a continuous query, triggered to drain-and-exit so it fits Free Edition's ephemeral compute.

Storage

Lakehouse tables
Delta Lake MERGE replaceWhere Unity Catalog

Why: ACID upserts make the medallion idempotent under at-least-once delivery; time-travel and atomic commits make replay safe.

Serving

Analytics surface
Databricks SQL BI / notebooks

Why: Gold exposes pre-aggregated daily grain, so dashboards read a stable, cheap surface instead of scanning raw events.

The boundary contract

Every message on heimdall.raw.business is one JSON envelope. Defined once in ingest/schemas.py, mirrored as a Spark StructType in the Silver job.

05

Delivery guarantees, end to end

The honest framing: the wire path is at-least-once, and idempotent writes collapse duplicates so the table state is effectively-once. We do not claim end-to-end exactly-once.

HopGuaranteeMechanism
Producer → Kafka at-least-once enable.idempotence, acks=all — no intra-session dupes
Kafka → Bronze exactly-once checkpoint + Delta atomic commit; failOnDataLoss=true
Bronze → Silver effectively-once MERGE on event_id — reprocessing updates in place
Silver → Gold idempotent replaceWhere overwrites only the bounded day partitions
Free Edition constraint, stated plainly: serverless compute has no always-on cluster, so every job uses trigger(availableNow=True) — drain the source, commit the checkpoint, exit. A Workflow schedule re-fires it every few minutes. Streaming correctness, batch economics.
